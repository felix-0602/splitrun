## 5. 项目现实勘察要求

开始拆 Plan.md 前，必须先用代码搜索和文件阅读确认当前项目现实，而不是只根据目标想象架构。

至少确认：

- **用户入口**：用户实际从哪个页面、组件、命令、快捷键或 API 进入。
- **调用链路**：入口如何调用前端 API client、后端 endpoint、service、数据库或外部服务。
- **新旧能力差异**：当前实现、目标实现、老接口/新接口、旧数据契约/新数据契约的差异。
- **消费字段**：前端或调用方实际消费了哪些响应字段，新增字段是否需要渲染、透传或忽略。
- **断点清单**：目标达成路径上已存在但尚未修复的问题，包括不是本轮代码造成的问题。
- **验收链路**：真实用户完成目标时应该观察到什么 UI、状态、日志、数据变化或副作用。

没有完成现实勘察，不允许进入 milestone 拆分；如果无法从代码中确认，必须在 Plan.md 和 Documentation.md 中记录未知项和阻塞影响。

## 6. 交付物规格

**契约同步原则**：改动函数签名、API 响应、环境变量、配置项、类型定义或启动命令时，对应的文档、类型和测试是**同一改动的组成部分**，必须在同一个 commit 中同步更新。这不是"扩 diff"——这是同一因果链条的完整闭环。判断标准：新人按旧文档/旧类型操作会不会跑不起来？会 = 没改完。

每个 milestone 交付物必须包含：

| 交付物 | 标准 |
|--------|------|
| **代码** | 通过 lint + type check + build，文件 ≤ 800 行，函数 ≤ 50 行 |
| **测试** | 关键路径必须有自动化测试；覆盖率阈值由本项目显式指定，未指定时不使用固定百分比作为硬门槛；不得为了让测试通过而删除关键断言、放宽真实业务约束或只测试实现细节 |
| **文档** | 文档必须按 Plan.md 的 milestone 同步更新；关键模块有 1 行职责说明，README 有运行方式，接口/数据契约/用户入口变化必须反映到对应文档 |
| **版本管理** | 每个可交付 milestone 必须说明版本影响：无版本变化 / patch / minor / major；涉及用户可见行为、API 契约、数据 schema、配置或部署方式变化时必须更新版本记录 |
| **数据** | 如有迁移脚本，必须可回滚 |

## 7. 质量维度

每个 milestone 标注**主导质量维度**（选 1 个），次要维度在括号标注。

| 维度 | 图标 | 关注点 |
|------|------|--------|
| **UX** 用户体验 | 🎨 | 交互流程、错误提示、加载状态、可访问性 |
| **Security** 安全 | 🛡 | 认证鉴权、输入校验、敏感数据、防注入 |
| **Performance** 性能 | ⚡ | 响应时间、资源占用、缓存策略、bundle 大小 |
| **Architecture** 架构 | 🏗 | 模块边界、接口设计、分层、可维护性 |
| **Module Depth** 模块深度 | 🧩 | 接口是否为有效测试面、浅模块集群信号、接缝是否有 ≥2 个适配器 |
| **Data Integrity** 数据完整 | 🔒 | 一致性、事务、幂等、防重复 |
| **Observability** 可观察 | 🔭 | 日志、错误追踪、健康检查、监控 |
| **DevOps** 部署运维 | 🚀 | CI/CD、部署、回滚、环境管理 |

## 8. 质量门禁解释

自动评估、测试数量、覆盖率和评分只能作为质量信号，不能替代项目目标、真实验收用例和人工 review。

- 评估 PASS 不等于生产可用；必须同时满足 Done When、关键路径验证和剩余风险记录。
- 分数提升必须对应真实风险下降；只为提高评分而增加表面字段、空 schema、空测试或形式化日志，不算有效交付。
- 测试应证明用户行为、系统契约、权限边界、错误路径和数据完整性；不得只验证"代码按当前实现返回了某个字段"。
- 如果为了稳定测试临时放宽断言、跳过场景或删除用例，必须在 Documentation.md 的已知问题或运行记录中写明原因和剩余风险。
- 文档和版本记录是交付物的一部分；代码通过但文档、契约说明或版本记录缺失时，milestone 不能标记完成。
