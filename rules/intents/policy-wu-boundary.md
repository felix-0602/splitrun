# policy/wu-boundary: Work Unit 文件边界

**防止什么危害**：执行一个 WU 时顺手改了不属于它的文件，造成不可预料的副作用。

**为什么存在**：Work Unit 的设计假设是"一块小任务，有明确的输入文件和产出文件"。`files_allowed` 是这个契约的执行层——如果你改了计划外的文件，(a) 其他并行 WU 可能同时在改同一文件，(b) 改动范围失控，(c) 验收测试无法验证。

**不遵守的后果**：
- 修 auth bug 顺手改了 UI → 引入新的 UI bug，但本 WU 的测试不覆盖 UI
- 两个 fork worker 同时改了同一个文件 → merge conflict 且没人知道谁改了什么
- 修复测试失败时改了业务逻辑而非测试 → 测试"绿了"但测的不是原来的行为了

**正确路径**：如果确实需要改边界外的文件 → 回 PLAN_STEP 扩边界或拆新 WU，不要偷偷越过。
